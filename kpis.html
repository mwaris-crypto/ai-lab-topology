LLM Inference KPIs, Explained

TTFT · TPOT · TPS · Total latency · Jitter — what each one means, how it's calculated, why it matters, and the network metric it mirrors.

One mental model. Serving an LLM answer is like delivering data over a network. TTFT is how long before the first byte arrives. TPOT / TPS is the bandwidth — how fast the rest streams. Total latency is the end-to-end transfer time. Jitter is how steady the stream is. Get those four right and you have a fast, smooth service.

Each metric in detail

TTFT  Time To First Token

What it is: the delay from sending the request to receiving the very first token of the reply.

What it shows: responsiveness — how quickly the system "starts talking." It's dominated by prefill (reading the whole prompt) plus any queue wait if the server is busy.

Why it matters: it's the lag the user actually feels before anything appears. In chat and agents it's the #1 driver of "feels fast." Long prompts and a loaded server both push it up.

How to calculate: TTFT = t(first token) − t(request sent). Includes network + queue + tokenize + prefill + generating that first token.

Network twin — Time To First Byte (TTFB): just like a web server's "time to first byte," TTFT is how long before the first piece of the response shows up. A page that's blank for 2 seconds feels broken — same with a chatbot.

TPOT  Time Per Output Token (a.k.a. average ITL)

What it is: the average time to produce each token after the first, during decode.

What it shows: the steady streaming speed — how fast words appear once the reply starts.

Why it matters: it sets how long a long answer takes. As a rule of thumb, humans read ~5–8 tokens/second, so a TPOT under ~150 ms/token (≈ 7+ tokens/s) feels real-time; slower than that and the user waits on the text.

How to calculate: TPOT = decode_time ÷ (N−1), where decode_time = latency − TTFT. It is the mean of all inter-token gaps.

Network twin — per-packet serialization delay: the steady time to push each packet onto the wire. TPOT is the steady time to emit each token.

TPS  Tokens Per Second (throughput)

What it is: how many tokens are generated per second. Two flavors: per-request TPS (one user's speed) and aggregate TPS (everything the server emits across all concurrent requests).

What it shows: per-request TPS = how fast one user is served; aggregate TPS = the server's total capacity.

Why it matters: aggregate TPS is the operator's capacity-and-cost number — "how many users/tokens can one GPU serve." Continuous batching raises aggregate TPS by sharing each weight read across several requests.

How to calculate: TPS = output_tokens ÷ decode_time. Per request it's ≈ 1000 ÷ TPOT(ms) (TPS and TPOT are inverses).

Network twin — bandwidth / throughput: bits-per-second on a link. Per-request TPS ≈ one flow's goodput; aggregate TPS ≈ total link throughput.

Latency  Total (end-to-end) request latency

What it is: the full time from sending the request to receiving the last token of the complete answer.

What it shows: total wait for the whole response.

Why it matters: it's the SLA for full completion. It dominates when the consumer needs the entire output before acting — non-streaming UIs, batch jobs, and multi-step agents where each step's latency compounds.

How to calculate: Latency = t(last token) − t(sent) = TTFT + decode_time = TTFT + (N−1) × TPOT. Measured client-side, so it includes network hops and queue wait — what the user really experiences.

Network twin — end-to-end transfer time: total time to fetch a file = connection setup + transferring all the bytes. Latency = TTFT + streaming all the tokens.

Jitter  Inter-token latency variation

What it is: the variability of ITL — how much the gap between tokens bounces around.

What it shows: smoothness. A low average TPOT can still feel choppy if some tokens stall; jitter exposes that.

Why it matters: consistency is a carrier-class requirement. Under load and batching, the scheduler interleaves requests and individual tokens get delayed, so production SLOs are written on the p95 (or p99) ITL, not the average.

How to calculate: jitter = stdev(ITL), or the tail form p95(ITL) − p50(ITL). Needs ≥3 tokens to be meaningful.

Network twin — packet delay variation (the classic "jitter"): in VoIP/video, uneven packet arrival makes calls choppy even when average latency is fine. Token jitter makes a stream stutter the same way.

A worked example (numbers that tie together)

One request: the first token arrives at 0.30 s; the model then generates 200 tokens, finishing at 4.30 s.

TTFT0.30 s − 0.00 s300 ms

As a network story: first byte in 300 ms (TTFB), then content streams at ~50 tokens/s (bandwidth), whole transfer done in 4.3 s (end-to-end), and the stream is fairly smooth (low jitter). Speed it up by lowering TTFT (shorter prompt / less queue) or raising TPS (faster GPU / batching); smooth it out by lowering p95 ITL.

Ranking by how critical they are

Criticality depends on the workload, but for an interactive, user-facing service (chat, agents — the carrier-class case), the usual order is:

  1. TTFT — the first impression. Nothing feels fast if the reply is slow to start.
  2. TPOT / per-request TPS — the streaming speed; must keep up with reading and not drag on long answers.
  3. Jitter (p95 ITL) — consistency; protects the experience under load, where SLOs live.
  4. Total latency — the full-completion SLA; becomes #1 for non-streaming and multi-step agent flows.
  5. Aggregate TPS — the operator's capacity/cost metric; critical for the business, less so for any single user.

How the ranking shifts by workload:

WorkloadMost critical → least
Interactive chat / agent (user waiting)TTFT > TPOT > jitter (p95 ITL) > latency > aggregate TPS
Multi-step agents / long-form (latency compounds)Total latency & TTFT > TPOT > jitter > aggregate TPS
Batch / offline generation (no human waiting)Aggregate TPS > total latency > TPOT > TTFT > jitter

How they all relate (one box)

ITLᵢ = t(token i) − t(token i−1)
TPOT = mean(ITL) = decode_time ÷ (N−1)
TPS = output_tokens ÷ decode_time ≈ 1000 ÷ TPOT(ms)
Latency = TTFT + decode_time = TTFT + (N−1) × TPOT
Jitter = stdev(ITL) or p95(ITL) − p50(ITL)

Definitions follow common industry usage (TTFT; TPOT, also called inter-token latency or ITL; throughput in tokens/s; end-to-end latency; jitter as ITL variation). Two honest conventions to know: (1) TPOT is sometimes computed as decode_time ÷ N instead of ÷ (N−1) — the difference is one token and is negligible for long outputs, but ÷ (N−1) is the precise "time per token after the first." (2) TPS is reported either per-request or aggregated across concurrent requests; always state which. All values are measured client-side, so latency includes network and queue time on purpose — that is what the end user experiences. Safe to publish: contains no addresses, hostnames, or credentials.
