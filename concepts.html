Plain-language explanation of every idea in the lab, with the best Read (📖), Watch (🎥) and Course (🎓) links. Each concept maps to a lab.

0. The foundation: how LLMs actually work

What it is. A large language model is a neural network trained to predict the next token (a chunk of text) given the previous tokens. "Chatting" is just the model repeatedly predicting the next token. Everything in inferencing is about doing that prediction fast, cheaply, and reliably.

Why it matters. Every metric you'll measure (TTFT, TPOT, VRAM) is a consequence of this one mechanism running on your GPU.

1. Tokens & tokenization Lab 1.6

What it is. The model never sees letters — it sees tokens, integer IDs for text chunks. "the" might be 1 token; "Agentic" might be 3. A tokenizer (using Byte-Pair Encoding) splits text into the model's fixed vocabulary. Rule of thumb: 1,000 tokens ≈ 750 words. Rare words and emoji cost more tokens because they get chopped into pieces.

Why it matters. Cost, context limits, and throughput are all measured in tokens — not words. This is the unit of everything downstream.

2. Inference KPIs: TTFT, TPOT, latency, throughput Labs 1.1, 4.x

What it is.

Why it matters. These are the language of carrier-class delivery. A fast TPOT with a slow TTFT still feels laggy; high throughput can come at the cost of per-user latency.

  • TTFT (Time To First Token) — gap between hitting Enter and the first token. Driven by prefill (processing your prompt). The metric users feel in chat.
  • TPOT (Time Per Output Token) — steady-state generation speed once it starts (tok/s). Driven by decode.
  • Request latency — total end-to-end time = TTFT + (TPOT × output tokens).
  • Throughput — total tokens/sec across all concurrent requests (a server-side metric).

3. Prefill vs. decode (the two phases) Lab 1.4

What it is. Prefill processes your entire prompt in one parallel pass — compute-bound, so it scales with prompt length and drives TTFT. Decode generates tokens one at a time, each needing the full model weights read from VRAM — memory-bandwidth-bound, so it sets your TPOT and is where "memory stalls" appear.

Why it matters. Explains why long prompts hurt TTFT but not per-token speed, and why your GPU can sit at 30% compute while still being the bottleneck (it's waiting on memory).

4. Quantization & "P & Q" Lab 1.2

What it is. Quantization stores model weights at lower precision (FP16 → 8-bit → 4-bit) to shrink VRAM use and speed up memory reads, at a small quality cost. GGUF is the file format llama.cpp/LM Studio use; Q4_K_M is a popular 4-bit variant (your Qwen weights at Q4_K_M ≈ 4.7 GB). "P & Q" = Parameters (model size, e.g. 7B) and Quantization.

Why it matters. On a 12 GB card, quantization decides which models even fit and how much VRAM is left for context and databases.

5. KV cache, context window & VRAM Lab 1.3

What it is. While generating, the model caches the "keys and values" of every prior token so it doesn't recompute them — the KV cache. It grows with context length × layers and lives in VRAM. The context window (Qwen reaches 128k) is how many tokens the model can "remember" at once. PagedAttention (vLLM) manages KV cache like OS virtual memory, cutting waste ~90%.

Why it matters. After weights, the KV cache is the #1 VRAM consumer and the hard limit on how long a context and how many concurrent users you can serve.

  • 📖 PagedAttention paper / vLLM blog
  • 📖 vLLM docs (KV cache, memory)

6. Sampling parameters Lab 1.5

What it is. Knobs that shape the output: temperature (randomness), top_p / top_k (how many candidate tokens to consider), max_tokens (length cap), stop sequences. Temperature/top_p/top_k change quality and variety with ~no latency cost; max_tokens directly drives latency and cost.

Why it matters. Distinguishing "quality knobs" from "cost knobs" is core to tuning a service to an SLO and budget.

  • 📖 How to generate text / sampling strategies (Hugging Face)

7. GPU telemetry & memory stalls Phase 2

What it is. nvidia-smi is the built-in GPU monitor (utilization, VRAM, power, temp). Its device-monitor mode (nvidia-smi dmon) shows compute (sm%) and memory (mem%) activity side by side — when compute is low while memory is high, the GPU is waiting for data = a memory stall. Decode is memory-bound, so stalls are normal and expected.

Why it matters. This is the "physics" layer — it explains why your numbers are what they are and where the real bottleneck lives.

8. Network forensics: TCP, QUIC, streaming Phase 3

What it is. Your Mac↔Windows API uses a long-lived TCP connection: one handshake, then a stream of tiny packets — each a token (token IDs are small integers, so downlink bandwidth is low even at fast speeds). Many web AI tools instead use QUIC (HTTP/3 over UDP). Wireshark lets you watch all of this on the wire.

Why it matters. It separates engine latency from network latency — what the user actually experiences across Wi-Fi or Tailscale.

9. Serving engines & continuous batching Phase 4

What it is. A desktop app (LM Studio) serves requests mostly one at a time. Production engines (vLLM, TGI, TensorRT-LLM, Ollama) use continuous (in-flight) batching — packing new requests onto the GPU the instant capacity frees up — plus PagedAttention. This is the single biggest throughput lever. Load testing (Locust, llmperf) drives concurrent traffic to find the latency "knee" and your capacity per GPU.

Why it matters. Carrier-class delivery is about serving many users within an SLO — that's an engine + batching + capacity-planning problem.

10. SLOs, the latency knee & capacity planning Lab 4.2–4.3

What it is. An SLO (Service Level Objective) is a target like "p95 TTFT < 500 ms." As you add load, latency stays flat then suddenly spikes — the knee — where queueing explodes. Capacity-per-replica = the max requests/sec you sustain below the knee while meeting the SLO.

Why it matters. It's the number every capacity, cost, and scaling decision is built on.

  • 📖 Google SRE Book — SLOs

11. RAG: embeddings & vector databases Phase 5 — your focus area

What it is. Retrieval-Augmented Generation grounds the LLM in your documents. Pipeline: split docs into chunks → an embedding model turns each chunk into a vector (a list of numbers capturing meaning) → store vectors in a vector database (ChromaDB, Qdrant) → at query time, embed the question, retrieve the nearest chunks by cosine similarity, and feed them to the LLM as context.

Why it matters. This is how you make the model an expert on your NextCloud files without retraining it — and your stated path to becoming a RAG expert.

12. Knowledge graphs & GraphRAG NEW — Phase 5B

What it is. A knowledge graph stores facts as nodes (entities: people, files, projects) connected by edges (relationships: "authored", "depends-on"). GraphRAG retrieves connected knowledge by traversing the graph, instead of (or alongside) plain vector similarity. Vector RAG finds chunks that sound similar; GraphRAG can answer multi-hop questions ("which documents by the author of X also reference Y?") and is more explainable. Microsoft's GraphRAG also clusters entities into "communities" and summarizes them.

Why it matters. Pure vector RAG misses relationships and struggles with "connect-the-dots" questions. Combining both (hybrid GraphRAG) is the current expert-level retrieval pattern — directly relevant to indexing a large, interconnected NextCloud corpus.

13. NextCloud + LLM (your home-cloud RAG lab) NEW — Phase 5C

What it is. NextCloud (your home cloud on a Raspberry Pi 4 + external SSD, holding your documents & pictures) ships a built-in Assistant with Context Chat — a RAG feature that indexes your NextCloud documents into a local ChromaDB vector store and grounds an LLM's answers in them. You point it at your Windows LLM via the LocalAI / OpenAI Integration app. The Pi serves files; the 3080 Ti does the embeddings + inference (the Pi is too weak for heavy ML).

Why it matters. This turns the whole lab into one real, end-to-end RAG system over your own home-cloud documents and pictures — the perfect hands-on path to RAG expertise.

14. Agents, tools & MCP Lab 6.1

What it is. An agent is an LLM that can act — call tools (read a file, run code, search) in a loop: think → act → observe → repeat. The Model Context Protocol (MCP) is an open standard for connecting models to tools/data via a client–server interface, so any MCP-aware model can use any MCP tool.

Why it matters. Moves you from "chatting with a box" to building systems that do things with your files and services.

  • 📖 Anthropic — Model Context Protocol (intro + docs)
  • 📖 Anthropic — Building effective agents

15. Orchestration: LangChain & LangGraph Lab 6.2–6.4

What it is. LangChain wires LLMs, tools, and retrievers into pipelines. LangGraph models an agent as a state graph — nodes (steps) and edges (transitions), with branching and loops. That's what lets you build a self-correcting Coder → Tester → Router loop: if the tester catches an error, the router sends the error back to the coder to fix. A2A (agent-to-agent) and multi-agent systems have specialized agents collaborate (e.g. a Coder + a Security Auditor).

Why it matters. Linear chains can't retry or branch; real autonomous workflows need state and cycles — exactly what your coding-agent labs use.

16. Vision models (VLMs) & multimodal Lab 5.3

What it is. A Vision-Language Model (e.g. llama3.2-vision) accepts images alongside text — image patches become tokens the model reasons over. Costs more VRAM and tokens than text-only.
