5. Advanced Inference Monitoring — LISO / LILO / SILO / SISO

Every request has an Input Sequence Length (ISL) and an Output Sequence Length (OSL). Pairing short/long for each gives four "shapes." Because GPU time and memory = model weights + prefill (reading the input) + decode (writing the output), each shape stresses the server differently. The shapes_test.py test sends controlled requests for all four shapes and measures the KPIs, so you can see the prefill-vs-decode trade-off on real numbers.

6. What the statistics mean (definitions + examples)

Percentile (p50 / p95 / p99)

A percentile pN is the value below which N% of your measurements fall — it describes the distribution, not just the middle.

  • p50 = the median. Half of requests are faster, half slower. The "typical" experience.
  • p95. 95% of requests are at or below this number; only the worst 5% are slower. "1 in 20 users sees this or worse."
  • p99. Only the worst 1% exceed it. "1 in 100." This is the tail — the unlucky requests.

Example. Of 100 requests, 95 start in ~200 ms but 5 spike to ~800 ms. The average (~230 ms) and even p95 (~230 ms) look fine — but p99 ≈ 800 ms reveals that 1 in 100 users waits four times longer. That hidden tail is exactly why production SLOs are written on p95/p99, not the average — they capture the worst case real users actually feel.

Standard deviation (how spread out / how consistent)

Standard deviation (std-dev, σ) measures how far the values typically sit from their average. Low σ = consistent; high σ = erratic. It's in the same units as the data (ms, tok/s…), so it's easy to read.

Example. Two streams both average 20 ms/token. Stream A's std-dev is 2 ms (every token ~18–22 ms — smooth). Stream B's std-dev is 15 ms (some tokens 5 ms, some 60 ms — choppy). Same average, very different feel. In this lab, jitter is literally the std-dev of inter-token latency — that's why a low average TPOT can still feel choppy if jitter (σ) is high.

Variance (the square of std-dev)

Variance is the average of the squared distances from the mean. Std-dev is just its square root: σ = √variance. Variance is in squared units (ms²), which is hard to picture, so people usually report std-dev — but variance is the underlying quantity, and it punishes big outliers more (because the distances are squared).

Worked example. Three token gaps: 18, 20, 22 ms → mean = 20.
Distances from mean: −2, 0, +2 → squared: 4, 0, 4.
Variance = (4 + 0 + 4) ÷ 3 = 2.67 ms². Std-dev = √2.67 ≈ 1.63 ms (a smooth stream).
Now make it choppy — gaps 5, 20, 35 → mean still 20, distances −15, 0, +15 → squared 225, 0, 225 → variance = 150 ms², std-dev ≈ 12.2 ms. Same average, far larger spread — the squaring makes the outliers dominate.

Quick rule of thumb: p50 = what most users get · p95/p99 = what your unluckiest users get (your SLO) · std-dev / jitter = how consistent it feels · variance = std-dev squared (same idea, squared units).
